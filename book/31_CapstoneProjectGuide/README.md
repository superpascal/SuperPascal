# Chapter 31: Capstone Project Guide

**Part of:** [SuperPascal — How to Think Like a Programmer](../ThinkLikeAProgrammerBook.md)

---

## Overview

This chapter guides you through creating a complete, polished project from start to finish. You'll learn how to plan, develop, polish, and present a capstone project that demonstrates everything you've learned.

**Learning Objectives:**
- Plan and propose a complete project
- Develop core systems systematically
- Create and integrate art and sound
- Polish and test your project
- Package and present your work

## Project Ideas

Here are some project ideas to inspire you:

### Game Projects
- **Platformer** — Jump and run game with multiple levels
- **Puzzle Game** — Match-3, block puzzle, or logic puzzle
- **Shooter** — Top-down or side-scrolling shooter
- **RPG** — Simple role-playing game with combat
- **Arcade Game** — Classic arcade-style game

### Interactive Projects
- **Math Visualizer** — Visualize mathematical concepts
- **Music Player** — Interactive music visualization
- **Sensor Dashboard** — Display sensor data with graphics
- **Educational Tool** — Teaching tool for a subject

### Hardware Projects
- **Robot Controller** — Control robot via GPIO
- **Weather Station** — Display weather data from sensors
- **Home Automation** — Control lights and devices
- **Data Logger** — Record and display sensor data

## Project Timeline

A typical capstone project timeline:

**Week 1-2: Planning**
- Define project scope
- Create proposal
- Design architecture
- Plan milestones

**Week 3-5: Core Development**
- Implement core systems
- Build basic gameplay
- Test functionality
- Iterate on design

**Week 6-7: Content Creation**
- Create art assets
- Add sound effects
- Create game content
- Integrate assets

**Week 8-9: Polishing**
- Fix bugs
- Optimize performance
- Playtest and refine
- Polish user experience

**Week 10: Presentation**
- Package project
- Create documentation
- Prepare demonstration
- Present work

---

## Success Criteria

A successful capstone project should:

✅ **Demonstrate Learning** — Show understanding of core concepts  
✅ **Be Complete** — Fully functional from start to finish  
✅ **Be Polished** — Well-tested and refined  
✅ **Be Documented** — Clear code and documentation  
✅ **Be Presentable** — Ready to show and share
